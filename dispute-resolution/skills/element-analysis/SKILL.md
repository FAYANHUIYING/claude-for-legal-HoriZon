# /element-analysis 要件分析表

法官按要件审案（要件审判九步法是各级法院推行的审理方法），所以最有效的备讼方式
就是抢在法官之前按要件审自己：每项请求拆成构成要件，每个要件问一句——**证据在哪、
够不够硬、对方怎么打**。本技能产出那张表，而表上的**空格比满格更值钱**。

读取实务画像与活跃事项（同插件其他技能的约定）；事项上下文存在时，证据清单与
事实优先取自 `matters/<slug>/`。

## 工作流

### 第 1 步 —— 确认立场与请求

- **立场**：原告/申请人（检验己方请求权成立）｜被告/被申请人（找对方要件的洞 +
  搭己方抗辩）｜中立评估（接案/和解决策用，两侧都做）。
- **诉讼请求**：逐项列明（本金、利息、违约金、赔偿、解除、返还……）。每项请求
  单独成表——不同请求的请求权基础和要件不同，混在一张表里会互相遮蔽缺口。

### 第 2 步 —— 识别请求权基础

按请求权基础检索的通常顺序逐类排查（合同上请求权 → 缔约过失 → 无因管理 →
物权请求权 → 不当得利 → 侵权）`[模型知识—需验证]`（学理通说的检索次序，
个案按实际排查）。

- **竞合时全部列出**，不要替律师挑。责任竞合（如违约与侵权，民法典 §186
  `[模型知识—需验证]`）时给一张**选择对比**：各路径的要件难度、举证负担、
  赔偿范围（可得利益/精神损害）、诉讼时效、管辖差异——选哪条是律师的决定，
  本技能给的是摆满选项的桌子。
- 每个请求权基础给**精确条文**：有检索集成（元典/北大法宝）先核验现行有效并标
  `[已验证—YYYY-MM-DD]`；没有就标 `[模型知识—需验证]`，绝不臆造条号。

### 第 3 步 —— 拆解构成要件

把选定（或并列）的请求权基础拆成构成要件。示例结构（违约损害赔偿）：
合同成立并生效 → 对方实施违约行为 → 己方受有损失 → 违约行为与损失有因果关系 →
不存在免责事由（消极要件，由对方举证但需预判）。

- 每个要件注明依据：法条、司法解释、指导性案例/公报案例的裁判规则（来源标签照常）。
- **要件拆法有学理分歧的**（如缔约过失的要件构成、因果关系的判断标准），把分歧
  写出来标 `[review]`，按对己方有利的拆法主张、按对己方不利的拆法备防——不要
  假装只有一种拆法。
- **举证责任逐要件标注**：谁主张谁举证为原则，法定举证责任倒置（如环境侵权的
  因果关系、医疗损害的特定情形 `[模型知识—需验证]`）逐项注明——举证责任站在
  哪边，直接决定空格是谁的麻烦。

### 第 4 步 —— 证据映射

对每个要件，从证据清单（事项文件 / 用户提供 / `case-intake` 拆案结果）逐份映射：

- 每格写：**证据编号 + 名称 + 关键页/段落 + 一句话证明内容**。引用要能按图索骥，
  "见合同"不合格，"证据 3《运输协议》第 2 页第 4.2 条"合格。
- 证明力备注：该证据的三性弱点（复印件无原件、形成时间存疑、关联性间接）——
  需要深度评估的，调 `/hongzhi-core:evidence-evaluation` 逐份做，结论回填本表。
- 一份证据可以支撑多个要件；一个要件最好有**两条以上独立来源**的证据（单点支撑
  的要件在质证中是脆的，标 🟡）。

### 第 5 步 —— 缺口检测（核心输出）

逐要件给覆盖度评级，**缺口清单放在全表之前**：

- 🔴 **空格**：要件无任何证据支撑——请求在此处会断
- 🟡 **单薄**：仅单点支撑 / 仅当事人陈述 / 三性有硬伤
- 🟢 **覆盖**：两条以上独立证据，三性无重大瑕疵

每个 🔴🟡 配**补证路径**：找谁取什么（业务往来方/银行流水/申请法院调查取证/
申请鉴定），以及取证的现实难度。**举证期限在台账上且临近的，本节顶部加横幅**，
并提议把补证节点记入 `/dispute-resolution:deadline-tracker`。

### 第 6 步 —— 对抗侧预演

- **原告侧运行时**：列对方可能的抗辩及其要件（诉讼时效经过、免责事由、过错相抵、
  已履行/抵销、合同无效/可撤销……），逐项标注我方需要的反证。
- **被告侧运行时**：上面的表反过来做——对方每个要件找最薄弱的一环集中打击
  （阻断一个要件即阻断该请求），同时为己方积极抗辩（时效、抵销、反诉）单独成表。

### 第 7 步 —— 输出

工作成果页眉 + ⚠️ 审查备注（来源/阅读范围/交你判断/时效）照插件共享护栏。

```
【请求 N：……】请求权基础：民法典 §XXX [来源标签]
| # | 构成要件 | 依据 | 举证责任 | 证据映射 | 覆盖度 | 缺口/弱点 |
|---|---|---|---|---|---|---|

缺口清单（按严重度）
🔴 [要件] —— 无证据。补证路径：……
🟡 [要件] —— 单点支撑（仅证据5）。建议：……

对抗侧预演
| 对方抗辩 | 抗辩要件 | 对方已有/可能的证据 | 我方反制 |
```

收尾决策树（按共享护栏格式）：起草文书（按要件组织 → 交接
`/dispute-resolution:civil-complaint-drafting` / `defense-statement` /
`agent-brief`）｜补证（出取证清单）｜上报（要件有 🔴 且补证无望时，评估
和解/撤诉选项给合伙人）｜入台账（举证/开庭节点 → `deadline-tracker`）。

## 与相邻技能的分工

- `evidence-argument-chain`（证据链映射）自下而上：从证据出发问"能证明什么"；
  本技能自上而下：从要件出发问"拿什么证明"。**两张表对不上的地方就是风险**——
  建议大案两个都跑，互为校验。
- 专利侵权的权利要求对照 → `/ip-legal:patent`；保险纠纷的要件分析含特有要件
  （保险利益、近因）→ 结合保单条款单独处理。

## 护栏

- **绝不臆造要件或条号。** 要件拆解的每条依据可溯源；拆法存疑标 `[review]`。
- 对用户陈述的"这个要件已经满足"先核证据再写 🟢——用户的确信不是证据。
- 覆盖度评级是初步法律判断，全表交律师定稿；严重度跨技能下限规则适用
  （本表 🔴 流转到文书技能时不得静默降级）。
