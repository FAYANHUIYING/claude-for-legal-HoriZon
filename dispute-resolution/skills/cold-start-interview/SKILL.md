# /cold-start-interview

1. 检查 `~/.claude/plugins/config/claude-for-legal-horizon/dispute-resolution/CLAUDE.md`。若带 `--check-integrations`，跳过访谈——只重跑第 0 部分的"接了什么？"检测，并在该配置路径重写 `## 可用集成` 表。探测时：只有当某 MCP 工具调用确实成功，才报告 ✓；已配置但未测试的连接器标 ⚪ 并附一行确认方法。**绝不仅凭 `.mcp.json` 声明就报告 ✓**——那会误导用户以为接好了，其实没有。
2. 运行下面的访谈：先第 0 部分（角色 + 执业设置 + 集成），再领域专属问题。
3. 索取种子文件（见下）。
4. 写入 `~/.claude/plugins/config/claude-for-legal-horizon/dispute-resolution/CLAUDE.md`，按需创建上级目录。

## 冷启动检查

读取 `~/.claude/plugins/config/claude-for-legal-horizon/dispute-resolution/CLAUDE.md`：
- **不存在** → 开始访谈。
- **包含 `<!-- SETUP PAUSED AT: -->`** → 向用户问好，提议从该小节继续。
- **包含 `[PLACEHOLDER]` 标记但无暂停注释** → 模板从未完成；提议重新开始，或从占位符起始处继续。
- **已填充（无占位符、无暂停注释）** → 已配置；除非带 `--redo`，否则跳过。

模板结构在 `${CLAUDE_PLUGIN_ROOT}/CLAUDE.md`——以它作为分节脚手架。把填好的实务画像写入配置路径，按需创建上级目录。

## 检查共享的所务画像

查找 `~/.claude/plugins/config/claude-for-legal-horizon/company-profile.md`。

- **若存在：** 读取它。一行确认："你是 [姓名]，[执业设置]，就职于 [律所/公司]，在 [管辖地] 执业。对吗？（或说'更新'来修改共享画像。）" 确认后跳过所务问题，直接进入本插件专属问题。
- **若不存在：** 你将是该用户配置的第一个插件。问所务问题（所名/业务/规模/管辖地/风险姿态/上报联系人），按本插件 `references/company-profile-template.md` 模板写入共享画像，并告诉用户："我已保存你的所务画像——其他法律插件会读取它并跳过这些问题。"

## 访谈开始前

以"先分叉"的开场白起头，3-4 行短句，先问速启还是完整：

> **`dispute-resolution` 面向处理商事诉讼与仲裁全流程的人**——拆案、大事记、起诉/答辩/上诉/再审/代理词、判决分析、可视化、律师函、和解。
>
> **2 分钟**：完成你的角色、执业设置、常代理立场（原告/被告）、主要法院/仲裁机构和集成检测。 **15 分钟**：再加上从你的定稿文书提取行文风格、证据编排习惯、本地法院/仲裁口径关注点和重大策略上报链。
>
> 速启还是完整？（随时可用 `/dispute-resolution:cold-start-interview --redo` 升级。）

**速启路径：** 只问第 0 部分和最关键的 1-2 个领域问题，其余以 `[DEFAULT]` 标记写入配置，并收尾说明：当某技能输出感觉不对劲时，通常就是该调的某个默认值——它会告诉你是哪个。

**完整路径：** 走完下面全部小节。

设置只从你的回答和你明确上传的文档构建画像——不读个人 Claude 历史、其他对话或主目录下的 CLAUDE.md。若对话上下文中已可见相关信息（如先前提到的所名），先问"要用它吗？"再使用。

## 访谈节奏

- **假设答案已经写在某处。** 凡超过一句话的内容，默认问法是"贴个链接或文档，或给我简短版"——别逼用户把已写好的东西重打一遍。
- **一轮不超过 2-3 个可作答的提问**（子项也算）。检验：用户能否不滚屏答完？
- **暂停与续接。** 开头告诉用户：说"暂停"即保存进度——写部分配置并在顶部加 `<!-- SETUP PAUSED AT: [小节名] -->`，未答字段标 `[PENDING]`。重跑时从停处继续，不重问已答的。
- **绝不写带静默缺口的配置。** 写入前回看：列出被跳过的问题，问"现在补几个，还是留作占位符？" 每个占位符都应是用户主动选择跳过的。
- **用户陈述的法律事实随到随核。** 用户给出的条号、期限、阈值、管辖地与你所知冲突时，写入配置前先说出来：「你说 X；我的理解是 Y——确认哪个写进画像？`[前提已标记—需核实]`」错误事实进了 CLAUDE.md 会传播进未来每份输出。

## 第 0 部分：谁在用，以及接了什么

#### 谁在用？

> 谁会日常使用本插件？（这决定每份输出的工作成果页眉。）
>
> 1. **律师/法律专业人员** → 页眉 `保密 — 律师工作成果 — 宏志律师事务所 · 依律师指导制作`
> 2. **非律师但有律师对接** 3. **非律师且无对接** → 页眉 `研究笔记 — 非法律意见 — 据以行事前须经执业律师审查`

若答案是 2 或 3，说一次：输出将表述为供律师审查的研究而非定论；在有法律后果的步骤（签署、发函、立案、回复机关）前会暂停并提示律师审查。这不是免责声明，是工具擅长什么（检索/组织/结构）与持牌律师对具体情形的判断之间的分界。

#### 执业设置

> 独立执业/小所（无层级） | 中大型律所 | 企业内部法务 | 政府/法律援助/公益 | 都不是（用你自己的话描述，我据实建画像而不硬塞框）

独立/小所 → 上报问题改问"何时请外聘律师或同事给第二意见"；中大型所/内部法务 → 问审批链与签字人；公益 → 问督导结构。

#### 接了什么？

> 本插件可对接：法律检索（元典/北大法宝）、飞书/IM、文档存储。让我检查你配了哪些——需要它们的功能会工作，没有的优雅回退到手工，而非静默失败。

**检查实际接通的，而非配置了的**：能测试就调一个简单 MCP 工具，成功才报 ✓；无从探测报 ⚪"已配置未验证"；未找到报 ✗ 并附接入方法与回退说明。日后接上了，重跑 `/dispute-resolution:cold-start-interview --check-integrations`。

把以上写入配置的 `## 使用者角色`、`## 可用集成` 小节。

## 领域专属问题

### 第 1 部分：立场与战场

> 1. **常代理立场**：原告/申请人为主，被告/被申请人为主，还是两者皆有？（影响每个文书技能的默认视角与攻防侧重。）
> 2. **主要战场**：常去哪些法院/仲裁机构（如上海各级法院、上海仲裁委、贸仲）？本地裁判口径是经验资产——后续可沉淀进 references 案库。
> 3. **案由集中度**：最常见的 3-5 类案由（买卖合同、货运代理、股权、借贷……）？我据此调整技能的默认检查清单。

### 第 2 部分：流程与上报

> 1. **立案/重大策略由谁拍板**？默认：商事诉讼/仲裁立案与重大策略总是上报合伙人——要调整吗？
> 2. **期限管理习惯**：举证/上诉/申请再审期限目前怎么跟踪（台账/日历/系统）？我会在每个输出里突出期限项。
> 3. **文书风格**：起诉状/答辩状/代理词有没有所内格式规范或模板？贴上来，文书技能以它为准而非通用格式。

## 种子文件

> 1. **3-5 份定稿诉讼文书**（起诉状/答辩状/代理词/上诉状各来一份最好）。我提取行文格式、论证组织方式、证据引用习惯。
> 2. **1-2 份打过的案子的大事记或证据清单**。我学习你的事实编排颗粒度。
>
> 粘贴、给路径，或"暂时跳过"（标 [LIMITED DATA]）。

## 撰写实务画像

按 `${CLAUDE_PLUGIN_ROOT}/CLAUDE.md` 的模板结构，把填好的画像写入 `~/.claude/plugins/config/claude-for-legal-horizon/dispute-resolution/CLAUDE.md`。凡基于少于足量文档构建的小节标 `[LIMITED DATA — 已审阅 N 份文档]`。

## 写完之后

**展示本插件能做什么**（具体清单，不是笼统模板）：

> - **先拆案再动笔**——材料拆解、时间轴、争议焦点、对方漏洞、补充材料清单。试：`/dispute-resolution:case-intake`
> - **案件大事记**——上传判决/证据材料按时间梳理。试：`/dispute-resolution:case-chronology`
> - **起诉/答辩/上诉/再审/代理词**——`/dispute-resolution:civil-complaint-drafting`、`defense-statement`、`appeal-petition`、`retrial-analysis`、`agent-brief`
> - **判决书拆解与救济路径**——`/dispute-resolution:judgment-analysis`
> - **诉讼可视化**——时间轴/法律关系/资金流向/流程图。试：`/dispute-resolution:litigation-visualization`
> - **律师函 / 和解协议**——`/dispute-resolution:lawyer-letter`、`/dispute-resolution:settlement`
>
> **我对你的第一个建议：** 拿一个在办案件的材料跑 `/dispute-resolution:case-intake`——它是其余一切文书技能的地基。
